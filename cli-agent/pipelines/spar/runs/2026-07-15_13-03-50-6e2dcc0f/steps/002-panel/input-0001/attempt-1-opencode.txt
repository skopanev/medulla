[out] Use it to chain agents into a repeatable workflow instead of driving them by hand.
[out] 
[out] This repository contains:
[out] 
[out] - `cli-agent/` — the runner, executors, scripts, and the `spar` workflow
[out] 
[out] ## Setup
[out] 
[out] ```bash
[out] # Install from GitHub
[out] pip install git+https://github.com/skopanev/medulla.git
[out] 
[out] # Install a bundled skill — copies the workflow AND provisions the runtime
[out] # (.medulla/medulla + .medulla/scripts are symlinked to the live install)
[out] medulla install-skill spar
[out] 
[out] # Run any workflow
[out] medulla -w <workflow>
[out] medulla -w <workflow>

[out] medulla --docker -w <workflow> --build \

[out] ## CLI Reference
[out] 
[out] `medulla` supports two invocation styles (the `medulla` command comes from pip; from a source checkout use the repo-root `./medulla` launcher).
[out] 
[out] - Workflow mode: `medulla -w <workflow> [options]`
[out] - Direct pipeline mode: `medulla <run|validate|graph> <pipeline.yaml> [options]`
[out] 
[out] ### All CLI keys
[out] 
[out] - `-w`, `--workflow <name>` - resolve `.medulla/workflows/<name>/pipeline.yaml` and run that installed workflow.
[out] - `run <pipeline.yaml>` - run a pipeline by explicit file path.
[out] - `validate <pipeline.yaml>` - validate a pipeline and exit without running stages.
[out] - `graph <pipeline.yaml>` - generate and open a graph for a pipeline file.
[out] - `--var KEY=VALUE` - override a pipeline variable; repeatable; values split on the first `=` only.
[out] - `--docker` - run through the Docker wrapper instead of directly on the host.
[out] - `--mount PATH` - Docker mode only; mount an existing extra directory **read-only** at `/workspace/<dirname>`.
[out] - `--mount-rw PATH` - Docker mode only; mount an existing extra directory **read-write** at `/workspace/<dirname>`.
[out] - `--host` - auto-start `.medulla/scripts/host-builder.sh`, the host bridge for native tools such as `xcodebuild`, `cargo`, `gradle`, `java`, and `maestro`.
[out] - `--stage STAGE` - skip to a specific stage (bypass earlier rounds; requires vars from a prior run or `--var`).
[out] - `--dry-run` - load/resolve the pipeline without executing stage commands.
[out] - `--verbose` - enable verbose logging and executor diagnostics.
[out] - `--build` - Docker mode only; force image rebuild via `scripts/docker.py` with `--no-cache`.
[out] - `graph` generates a graph file for the pipeline.
[out] - If neither `-w` nor a valid `<command> <pipeline>` pair is provided, `medulla` exits with usage help.
[out] 
[out] ### Common command patterns
[out] 
[out] ```bash
[out] # Run an installed workflow
[out] medulla -w <workflow>
[out] 
[out] # Run a workflow with variables
[out] medulla -w <workflow> \
[out]   --var KEY=VALUE \
[out]   --var OTHER_KEY=OTHER_VALUE
[out] 
[out] # Dry-run a workflow locally
[out] medulla -w <workflow> --dry-run --verbose
[out] 
[out] # Validate a raw pipeline file
[out] medulla validate path/to/pipeline.yaml
[out] 
[out] # Generate a graph for a raw pipeline file
[out] medulla graph path/to/pipeline.yaml
[out] 
[out] # Run in Docker with extra mounts
[out] medulla -w <workflow> --docker --build \
[out]   --mount ../readonly-dir \
[out]   --mount-rw ../writable-dir
[out] ```
[out] 
[out] ### Docker wrapper details
[out] 
[out] When `--docker` is used, the wrapper additionally supports these behaviors:
[out] 
[out] - Image name comes from `MEDULLA_IMAGE` or defaults to `medulla:latest`.
[out] - Dockerfile resolution prefers `workflows/<workflow-root>/Dockerfile`, then falls back to the root `Dockerfile`.
[out] - Local config mounts are added automatically when present: `~/.claude`, `~/.codex`, `~/.gemini`, opencode config/auth, `~/.gitconfig`, and the host-builder bridge at `/tmp/medulla-bridge`.
[out] 
[out] Forwarded environment variables:
[out] 
[out] - `ANTHROPIC_API_KEY` - Anthropic credential.
[out] - `OPENAI_API_KEY` - OpenAI credential.
[out] - `ZHIPU_API_KEY` - Zhipu credential.
[out] - `GEMINI_API_KEY` - Gemini credential.
[out] - `GOOGLE_API_KEY` - Google API credential.
[out] - `GOOGLE_CLOUD_PROJECT` - Google Cloud project id.
[out] - `GOOGLE_APPLICATION_CREDENTIALS` - path to Google credentials.
[out]

[out] ## Workflow Structure
[out] 
[out] A workflow is a directory with a `pipeline.yaml` entry point plus any prompts/scripts it references.
[out] 
[out] Typical layout:
[out] 
[out] ```text
[out] workflows/<workflow>/
[out] ├── pipeline.yaml
[out] ├── prompts/
[out] └── scripts/
[out] └── ... any other fodlers you need
[out] ```
[out] 
[out] Minimal pipeline shape:
[out] 
[out] ```yaml
[out] version: "0.1"
[out] starting: first-stage
[out] round_timeout: 1500 # global timeout per stage
[out] 
[out] vars: # global variables available to all stages
[out]   KEY: default
[out] 
[out] fallback_runner: # global fallback runner if a stage fails
[out]   llm: claude-code
[out]   model: sonnet
[out] 
[out] stages:
[out]   first-stage:
[out]     runner:
[out]       llm: claude-code
[out]       model: sonnet
[out]       prompt: |
[out]         Do the task.
[out]     on_signal:
[out]       completed: __exit__ # transition to exit stage
[out]       failed: stage-name # transition to a named stage
[out]       default: __exit__
[out] ```
[out] 
[out] Main top-level fields:
[out] 
[out] - `version` - pipeline format/version label.
[out] - `starting` - name of the first stage.
[out] - `round_timeout` - default timeout budget for stage runs.
[out] - `vars` - default pipeline variables.
[out] - `fallback_runner` - optional pipeline-wide fallback runner if a stage runner fails.
[out] - `stages` - map of stage definitions.
[out] 
[out] ### Stage shape
[out] 
[out] Each stage is a named block under `stages`.
[out] 
[out] ```yaml
[out] some-stage:
[out]   max_iterations: 3
[out]   runner:
[out]     shell: ./scripts/do-work.sh
[out]     timeout: 60
[out]   fallback_runner:
[out]     llm: opencode
[out]     model: openai/gpt-5.4
[out]   on_signal:
[out]     ready: next-stage
[out]     failed: __exit__
[out]     default: __exit__
[out] ```
[out] 
[out] Stage fields:
[out] 
[out] - `runner` - required; defines how the stage executes.
[out] - `finally` - optional cleanup runner; executes after every stage exit (success, failure, timeout, on_max) before transitioning.
[out] - `fallback_runner` - optional stage-local fallback.
[out] - `on_signal` - required; transition table keyed by emitted signal name.
[out] - `max_iterations` - optional safety cap for repeated returns to the same stage.
[out] 
[out] ### Runner types
[out] 
[out] A stage runner must define exactly one of these:
[out] 
[out] - `shell` - run a shell command.
[out] - `llm` - run an LLM executor.
[out] - `loop` - iterate over a list and run an inner runner for each item.

[out] ### Variables and templating
[out] 
[out] Variables come from three places:
[out] 
[out] - `vars` in `pipeline.yaml`
[out] - CLI overrides from `--var KEY=VALUE`
[out] - runtime `<signal:var>` emissions from stages
[out] 
[out] Templates supported inside prompts and shell commands:
[out] 
[out] - `{{var:KEY}}` - insert variable value.
[out] - `{{var:KEY:-default}}` - insert variable with default fallback.
[out] - `{{file:path/to/file}}` - inline file contents.
[out] - `{{__list_item__}}` - current loop item identifier/path.
[out] - `{{__item__}}` - fetched content for the current loop item.
[out] 
[out] Variable signals look like this:
[out] 
[out] ```xml
[out] <signal:var key="TARGET_DIR">backend</signal:var>
[out] ```
[out] 
[out] That stores `TARGET_DIR=backend` and makes it available to later stages.
[out] 
[out] ### Calling LLMs
[out] 
[out] LLM stages use `runner.llm` plus an optional `model`.
[out] 
[out] Examples:
[out] 
[out] - `llm: claude-code`
[out] - `llm: claude-code` with `model: sonnet`
[out] - `llm: opencode` with `model: openai/gpt-5.4`
[out] - `llm: gemini`
[out] 
[out] The `prompt` field is the full rendered prompt sent to that executor.

[out] ### Signals and transitions
[out] 
[out] Stages communicate by emitting XML-like signals in stdout/stderr output:
[out] 
[out] ```xml
[out] <signal:ready>done</signal:ready>
[out] <signal:failed>something broke</signal:failed>
[out] <signal:update>progress message</signal:update>
[out] ```
[out] 
[out] `on_signal` maps those signals to the next action.
[out] 
[out] Simple transition:
[out] 
[out] ```yaml
[out] on_signal:
[out]   ready: next-stage
[out]   failed: __exit__
[out]   default: __exit__
[out] ```
[out] 
[out] Transition with options:
[out] 
[out] ```yaml
[out] on_signal:
[out]   approved:
[out]     stage: next-stage
[out]     reset_iterations: true
[out] ```
[out] 
[out] Transition with hook steps before moving on:
[out] 
[out] ```yaml
[out] on_signal:
[out]   approved:
[out]     - runner:
[out]         shell: python3 scripts/post_process.py
[out]     - stage: next-stage
[out] ```
[out] 
[out] Special targets:
[out] 
[out] - `__exit__` - stop the pipeline.
[out] - `__next_item__` - move to the next loop item.
[out] 
[out] Special signals:
[out] 
[out] - `default` - used when nothing matched.
[out] - `loop_done` - required for loop stages when iteration completes.
[out] - `on_max` - optional route when `max_iterations` is exceeded.
[out] - `update` - progress only; does not transition.
[out] - `var` - stores variables; does not transition.
[out] 
[out] ### Fallbacks
[out] 
[out] Fallbacks let the runner retry a failed stage with another runner.
[out] 
[out] - pipeline-level `fallback_runner` applies globally
[out] - stage-level `fallback_runner` overrides it for one stage
[out] 
[out] Use this when the main executor is flaky, overloaded, or you want a second model as backup.

[out] ### Host Builder
[out] 
[out] Runs on Mac, bridges tools to Docker container via `/tmp/medulla-bridge`.
[out] 
[out] ```bash
[out] .medulla/scripts/host-builder.sh
[out] ```
[out] 
[out] Protocol:
[out] 
[out] - Request ID protocol: `id:<pid.counter> <cmd>` -> response/exit_code in `response.<id>` / `exit_code.<id>` (eliminates race conditions)
[out] - Normal: `tool args` -> wait -> response + exit_code
[out] - Background: `bg:command` -> returns PID immediately
[out] - Kill: `kill:<PID>` -> terminates background process
[out] - On shutdown: kills all tracked BG_PIDS (firebase, emulators)
[out] 
[out] Bridged tools: java, gradle, xcodebuild, maestro, cargo.
